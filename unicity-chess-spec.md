# Unicity Chess — P2P Chess on Sphere

## Overview

A peer-to-peer chess game hosted on GitHub Pages, played inside Unicity Sphere as an iframe agent. Players connect via **Sphere Connect**, challenge opponents by nametag (`@alice`), and exchange moves over Sphere DMs (encrypted Nostr messages handled by Sphere). Fixed 10 UCT wager per game, managed through the `@unichess` escrow nametag.

## Identity & Wallet Connection

- Each player is identified by their **Unicity nametag** (e.g., `@alice`).
- The game connects to Sphere wallet via the **Connect protocol** using `ConnectClient`.
- Three transport priorities (same pattern as Boxy-Run and sphere-sdk-connect-example):
  1. **Iframe** (game inside Sphere) — `PostMessageTransport.forClient()`
  2. **Extension** (Sphere browser extension) — `ExtensionTransport.forClient()`
  3. **Popup** (standalone) — opens `https://sphere.unicity.network/connect?origin=...`
- On successful connect, the game receives a `PublicIdentity`: `{ nametag, l1Address, directAddress, chainPubkey }`.
- The game **never** touches private keys — all signing and messaging is delegated to Sphere.

### Required Permissions

Request these permission scopes during the connect handshake to minimize per-action approval modals:

- `identity:read` — read player identity
- `balance:read` — check UCT balance before deposit
- `comms:read` — receive DMs (game messages)
- `comms:write` — send DMs (game messages) without per-message approval
- `intent:send` — send UCT deposits
- `events:subscribe` — subscribe to real-time DM events

### Session Management

- Store session ID in `sessionStorage` (key: `unichess-session`)
- On page load, attempt silent auto-connect with `silent: true` and saved session
- On popup close (P3 mode), poll `popupWindow.closed` and treat as disconnect

## Message Protocol

All game messages are sent as **Sphere DMs** via `client.intent('dm', ...)`. Messages use a concise single-line text format (not JSON) to keep DM content readable and lightweight.

### Format

```
uc1:<gameId>:<action>[:<param1>:<param2>:...]
```

- `uc1` — protocol identifier + version (unicity chess v1)
- `gameId` — 8-character hex string (first 8 chars of a UUID v4, generated by challenger)
- `action` — short action code
- Remaining fields are action-specific, colon-separated

### Actions

#### `ch` — Challenge

```
uc1:<gameId>:ch:<color>:<timeMinutes>:<gameUrl>
```

- `color` — `w` (challenger plays white), `b` (black), or `r` (random)
- `timeMinutes` — `3`, `5`, or `10`
- `gameUrl` — HTTPS link to game page with `?game=<gameId>` parameter

Example: `uc1:a1b2c3d4:ch:w:5:https://user.github.io/unicity-chess/?game=a1b2c3d4`

#### `ok` — Accept challenge

```
uc1:<gameId>:ok
```

#### `no` — Decline challenge

```
uc1:<gameId>:no
```

#### `mv` — Move

```
uc1:<gameId>:mv:<san>:<clockMs>
```

- `san` — Standard Algebraic Notation (e.g., `e4`, `Nf3`, `O-O`, `Qxd7+`)
- `clockMs` — remaining time in milliseconds for the player who just moved

Example: `uc1:a1b2c3d4:mv:e4:295320`

#### `rs` — Resign

```
uc1:<gameId>:rs
```

#### `do` — Draw offer

```
uc1:<gameId>:do
```

#### `da` — Draw accept

```
uc1:<gameId>:da
```

#### `dd` — Draw decline

```
uc1:<gameId>:dd
```

#### `hb` — Heartbeat

```
uc1:<gameId>:hb:<clockMs>
```

Sent every 30 seconds by the player whose turn it is NOT (the waiting player), so the active player knows the opponent is still connected. `clockMs` is the sender's remaining time.

#### `ab` — Abort

```
uc1:<gameId>:ab
```

Either player can abort before both players have made at least one move each.

#### `go` — Game over claim

```
uc1:<gameId>:go:<result>:<reason>
```

- `result` — `w` (white wins), `b` (black wins), `d` (draw)
- `reason` — `checkmate`, `resign`, `timeout`, `stalemate`, `agreement`, `repetition`, `50move`, `material`, `disconnect`

Sent when a client detects a terminal condition (checkmate, timeout, etc.) to notify the opponent.

### Sending & Receiving

```typescript
// Send a game message
async function sendGameMessage(opponentNametag: string, message: string) {
  await client.intent('dm', {
    to: opponentNametag,   // e.g., '@bob'
    message,               // e.g., 'uc1:a1b2c3d4:mv:e4:295320'
  });
}

// Receive game messages (real-time via event subscription)
client.on('message:dm', (data: DirectMessage) => {
  if (!data.content.startsWith('uc1:')) return; // Not a chess message
  const parts = data.content.split(':');
  const gameId = parts[1];
  const action = parts[2];
  // Route to handler...
});

// Load missed messages on reconnect
const history = await client.query('sphere_getMessages', {
  peerPubkey: opponentPubkey,
  limit: 100,
});
// Filter for uc1: messages and replay
```

### Filtering Chess Messages

Chess protocol messages start with `uc1:` and are easily distinguished from regular DMs. The game only processes messages matching the current `gameId`. Messages from unknown game IDs with action `ch` (challenge) are treated as incoming challenges.

## Game Flow

### 1. Deposit Phase

Both players must deposit 10 UCT to `@unichess` before the game starts.

```
Player A                                        Player B
   |                                                |
   |-- connects Sphere wallet                       |
   |-- enters @bob as opponent                      |
   |-- selects color + time control (3/5/10 min)    |
   |-- deposits 10 UCT to @unichess                 |
   |-- generates gameId (8-char hex)                 |
   |-- sends "ch" DM to @bob ---------------------->|
   |                                                |-- receives DM (or clicks gameUrl link)
   |                                                |-- sees incoming challenge
   |                                                |-- deposits 10 UCT to @unichess
   |<------------------ sends "ok" DM --------------|
   |                                                |
   |   === GAME BEGINS ===                          |
```

- Challenger deposits **before** sending the challenge DM.
- Accepter deposits **before** sending the accept DM.
- If no `ok` received within **5 minutes**, the challenge expires. Challenger's deposit is refunded via faucet.
- The deposit uses `client.intent('send', { to: '@unichess', amount: '10', coinId: 'UCT', memo: 'unichess:<gameId>' })`.
- The memo field ties the deposit to the specific game for audit purposes.

### 2. Play Phase

```
White                                           Black
   |                                                |
   |-- makes move on board (drag & drop)            |
   |-- chess.js validates the move locally           |
   |-- sends "mv" DM with SAN + remaining clock --->|
   |                                                |-- receives DM
   |                                                |-- chess.js validates SAN
   |                                                |-- if valid: apply, render, start own clock
   |                                                |-- if invalid: show warning
   |                                                |
   |<-------------- sends "mv" DM ------------------|
   |                                                |
   |   (waiting player sends "hb" every 30s)        |
   |                                                |
   |   ... continues until game ends ...            |
```

### 3. End Conditions

| Condition | Trigger | Result |
|---|---|---|
| Checkmate | chess.js `isCheckmate()` | Winner takes 20 UCT |
| Stalemate | chess.js `isStalemate()` | Draw — 10 UCT each |
| Resignation | Player sends `rs` | Opponent wins — 20 UCT |
| Draw by agreement | `do` followed by `da` | Draw — 10 UCT each |
| Threefold repetition | chess.js `isThreefoldRepetition()` | Auto-draw — 10 UCT each |
| 50-move rule | chess.js `isDraw()` | Auto-draw — 10 UCT each |
| Insufficient material | chess.js `isInsufficientMaterial()` | Auto-draw — 10 UCT each |
| Timeout | Player's clock reaches 0 | Opponent wins — 20 UCT |
| Abort | `ab` sent before move 2 | No result — 10 UCT refunded each |
| Disconnect | No heartbeat/move for full game duration after last heartbeat | Opponent wins — 20 UCT |

### 4. Payout Phase

On game end, the game calls the faucet API to distribute funds from `@unichess`:

```typescript
// Winner takes all
await fetch('https://faucet.unicity.network/api/v1/faucet/request', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    unicityId: winnerNametag,  // e.g., '@alice'
    coin: 'unicity',
    amount: 20,
  }),
});

// Or on draw — 10 UCT to each
// Two separate faucet requests, 10 each
```

## Game State Management

### In-Memory State Only

Game state is held **in memory only** (React state). If a player closes the browser tab or navigates away, they forfeit the game. The opponent detects this via missing heartbeats and claims the win after the disconnect timeout.

### State Object

```typescript
interface GameState {
  gameId: string;
  status: 'idle' | 'depositing' | 'challenging' | 'awaiting-accept' | 'playing' | 'ended';
  myColor: 'white' | 'black';
  chess: Chess;                   // chess.js instance
  moveHistory: MoveRecord[];
  myClockMs: number;              // remaining time in milliseconds
  opponentClockMs: number;
  timeControlMinutes: 3 | 5 | 10;
  result: GameResult | null;
  opponent: {
    nametag: string;              // e.g., '@bob'
    pubkey: string;               // chainPubkey from PublicIdentity
  };
  drawOfferedBy: 'me' | 'opponent' | null;
  lastHeartbeatAt: number;        // Unix timestamp ms of last received hb/mv
  myDepositDone: boolean;
  opponentDepositDone: boolean;
}

interface MoveRecord {
  san: string;
  clockMs: number;
  timestamp: number;              // Unix timestamp ms when move was made
}

interface GameResult {
  outcome: 'white-wins' | 'black-wins' | 'draw' | 'aborted';
  reason: 'checkmate' | 'resign' | 'timeout' | 'stalemate' | 'agreement'
        | 'repetition' | '50move' | 'material' | 'abort' | 'disconnect';
}
```

## Clock / Time Control

- Three presets: **3 minutes**, **5 minutes**, **10 minutes** (no increment).
- Clocks use millisecond precision internally, displayed as `M:SS` or `S.s` (under 10 seconds).
- All timestamps are **Unix UTC milliseconds** (`Date.now()`).
- The active player's clock counts down locally via `requestAnimationFrame` for smooth display.
- When a player makes a move, they include their `clockMs` in the `mv` message.
- The receiver sets the opponent's clock to the received value.
- If a player's clock reaches 0, the game ends immediately — the client sends a `go` (game over) message with reason `timeout`.

### Clock Anti-Cheat

- Clamp received clock values: the opponent's reported time cannot be **higher** than what it was at their last move minus a reasonable minimum (100ms). If a received clock is higher than expected, use the locally tracked value instead.

---

## Heartbeat & Disconnect Detection

- The **waiting player** (player whose turn it is NOT) sends a `hb` message every **30 seconds** with their remaining clock time.
- Each client tracks `lastHeartbeatAt` — the timestamp of the last received `hb` or `mv` message.
- **Disconnect timeout**: If no `hb` or `mv` is received for the **full game duration** (e.g., 5 minutes in a 5-minute game) after the last received message, the remaining player can claim a win.
  - The client shows a "Claim win — opponent disconnected" button.
  - Clicking it sends a `go` message with reason `disconnect` and triggers the payout.
- Heartbeats also serve as a clock sync mechanism — both players see each other's remaining time.

## Wager Flow

Fixed 10 UCT per game, managed through `@unichess` escrow nametag.

### Deposit

```typescript
const depositResult = await client.intent('send', {
  to: '@unichess',
  amount: '10',
  coinId: uctCoinId,  // Resolved dynamically via sphere_getBalance
  memo: `unichess:${gameId}`,
});
```

### Payout

The game calls the faucet API directly (same endpoint as Boxy-Run):

| Outcome | Payout |
|---|---|
| Win | Winner receives 20 UCT from faucet |
| Draw | Each player receives 10 UCT from faucet |
| Abort (< 2 moves) | Each player receives 10 UCT refund from faucet |
| Challenge expired | Challenger receives 10 UCT refund from faucet |

### UCT Coin ID Resolution

The UCT coin ID (hex hash) is resolved dynamically at runtime from the balance response, not hardcoded:

```typescript
const assets = await client.query('sphere_getBalance');
const uct = assets.find((a: any) => a.symbol === 'UCT');
const uctCoinId = uct?.coinId;
```

Fallback to known hex ID if balance is empty: `455ad8720656b08e8dbd5bac1f3c73eeea5431565f6c1c3af742b1aa12d41d89`

## Move Validation

Both sides validate every move independently. On receiving a `mv` message:

1. Check `gameId` matches the current game.
2. Check it is the opponent's turn (based on local chess.js state).
3. Attempt to apply the SAN move via `chess.move(san)`.
4. If the move is legal — accept, update state, render board, start own clock.
5. If the move is illegal — **do not apply**. Show a warning: "Invalid move received from opponent." Do not auto-forfeit.

FEN is **not** included in move messages (keeping the protocol concise). Both clients derive FEN independently from chess.js. If states diverge (extremely unlikely with valid move validation), the game is in an unrecoverable state and either player can resign.

## Error Handling

| Scenario | Behavior |
|---|---|
| Wallet not connected | Show Connect button. Disable all game actions. |
| Sphere disconnected mid-game | Show "Reconnecting…" banner. Attempt silent re-connect. Game timer pauses for local player. |
| Insufficient UCT balance | Show "Need 10 UCT to play" message. Disable challenge/accept. |
| Deposit fails | Show error. Do not send challenge/accept. Allow retry. |
| Invalid move received | Log warning. Do not apply. Show notification to user. |
| Opponent disconnects | Track via missing heartbeats. Show countdown. Offer "Claim win" after timeout. |
| Faucet payout fails | Log error. Show "Payout pending — contact support" message. Retry once. |
| DM send fails | Retry up to 3 times with 1s delay. If all fail, show error banner. |
